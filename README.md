# integration_demo

A small demonstration of a Rust world-state library compiled to WebAssembly and
driven from a SvelteKit frontend. Three characters share a scene in a teahouse;
the user picks actions, the Rust side advances the world, and the frontend
renders the result.

The point of the demo is the plumbing — how Rust types, JS bindings, and
TypeScript interfaces are generated and kept in sync across the WASM boundary.

## How it links up

```
  ┌──────────────────────────────┐         ┌───────────────────────────────┐
  │  crates/world  (Rust, cdylib)│         │  src/  (SvelteKit, TS)        │
  │                              │         │                               │
  │  #[wasm_bindgen] impl World  │         │  import init, { World }       │
  │     getState() -> WorldState │         │      from 'world';            │
  │     applyAction(id) -> Msg   │ ──────► │  await init();                │
  │                              │  wasm   │  const w = new World();       │
  │  #[derive(Tsify, Serde)]     │  +js    │  w.getState(); // typed       │
  │     Character, Message, …    │         │                               │
  └──────────────────────────────┘         └───────────────────────────────┘
             │                                         ▲
             │  wasm-pack build --target web           │
             ▼                                         │
  ┌──────────────────────────────┐                     │
  │  crates/world/pkg/           │                     │
  │     world.js                 │  file: dependency   │
  │     world.d.ts (from Tsify)  │  symlinked into     │
  │     world_bg.wasm            │  node_modules/world │
  └──────────────────────────────┘ ────────────────────┘
```

The pieces:

- **Rust crate (`crates/world/`)** — a `cdylib` library containing all world
  state and logic. Every struct that crosses the boundary derives
  `Serialize`, `Deserialize`, and `Tsify` with `#[tsify(into_wasm_abi,
  from_wasm_abi)]`. The `World` handle is exposed as a `#[wasm_bindgen]`
  class with two methods: `getState()` and `applyAction(id)`.

- **`wasm-pack build --target web`** compiles the crate to `crates/world/pkg/`,
  emitting `world_bg.wasm`, a JS ES-module loader (`world.js`), and a
  TypeScript declaration file (`world.d.ts`) generated by `tsify`. The
  declarations describe the Rust structs as TS `interface`s, so the frontend
  gets full type coverage for everything returned from Rust.

- **`package.json`** declares `"world": "file:./crates/world/pkg"`. npm creates
  a symlink at `node_modules/world` pointing at the pkg directory, so rebuilds
  of the WASM are picked up without reinstalling.

- **`vite.config.ts`** loads `vite-plugin-wasm` and
  `vite-plugin-top-level-await` so the async WASM module can be imported like
  any other ES module. `optimizeDeps.exclude: ['world']` keeps Vite from
  trying to pre-bundle the WASM entry, and `resolve.preserveSymlinks: true`
  keeps path resolution stable across the symlink.

- **SvelteKit frontend (`src/routes/+page.svelte`)** imports the default
  `init` (to instantiate the WASM module) plus the `World` class and the
  generated type aliases. `src/routes/+layout.ts` sets `ssr = false` because
  the WASM module is browser-only.

### Data flow for one click

1. User clicks an action button in the browser.
2. Svelte calls `world.applyAction(id)`. The `id` string is marshalled into
   WASM linear memory by `wasm-bindgen`.
3. Rust mutates the `WorldState` (appends a message, shifts a character's
   emotion, advances the action cycle), then returns the new `Message`.
4. The returned value is serialized by `tsify` / `serde-wasm-bindgen` directly
   into a JS object — no JSON string intermediary — typed on the JS side as
   `Message`.
5. Svelte reads the fresh `WorldState` via `world.getState()` and re-renders.

## Prerequisites

- Rust toolchain with the `wasm32-unknown-unknown` target (`wasm-pack` will
  install it on first build if missing).
- [`wasm-pack`](https://rustwasm.github.io/wasm-pack/) — install with
  `cargo install wasm-pack`.
- Node.js 18+ and npm.

## Build and run

First build:

```sh
wasm-pack build crates/world --target web --out-dir pkg
npm install
npm run dev
```

Open <http://localhost:5173/>.

The `wasm` script in `package.json` is a shorthand for the first step:

```sh
npm run wasm    # rebuild the Rust crate
npm run dev     # SvelteKit dev server
```

After the initial `npm install`, subsequent changes to the Rust crate only
need `npm run wasm` — the `node_modules/world` symlink picks up the new
output and Vite hot-reloads.

## Production build

```sh
npm run build     # prerendered static site in build/
npm run preview   # serve the built site locally
```

The site is built with `@sveltejs/adapter-static` as a single-page app
(`fallback: 'index.html'`), so it can be served from any static host.

## What the Rust API looks like

```rust
#[wasm_bindgen]
impl World {
    #[wasm_bindgen(constructor)]
    pub fn new() -> World;

    #[wasm_bindgen(js_name = getState)]
    pub fn get_state(&self) -> WorldState;

    #[wasm_bindgen(js_name = applyAction)]
    pub fn apply_action(&mut self, action_id: String) -> Message;
}
```

And the generated TypeScript (`world.d.ts`) mirrors it exactly:

```ts
export interface Character { id: string; name: string; bio: string; emotion: string; }
export interface Message   { id: number; sender: string; text: string; system: boolean; }
export interface Action    { id: string; label: string; }
export interface WorldState {
    characters: Character[];
    messages: Message[];
    actions: Action[];
    cycle: number;
}

export class World {
    constructor();
    getState(): WorldState;
    applyAction(action_id: string): Message;
}
```

## Notes on the boundary

- `Vec<Struct>` is not directly returnable through `wasm-bindgen` as a method
  return type. The demo works around this by returning the whole `WorldState`
  (whose vectors are serialized by `tsify`) rather than exposing individual
  `Vec`-valued getters.
- `console_error_panic_hook` is installed in the `World` constructor so any
  Rust panic during development surfaces as a readable stack trace in the
  browser console.
- Serialization across the boundary uses `serde-wasm-bindgen`, which moves
  values as JS objects directly — there is no `JSON.parse(JSON.stringify(…))`
  round-trip.
